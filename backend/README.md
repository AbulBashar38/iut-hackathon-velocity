# IoT Office Monitoring — Backend

Backend for the **IoT Office Monitoring System**. There are no real IoT devices:
this service **simulates** an office full of devices, calculates power usage,
raises alerts, and streams everything to clients in real time over **Socket.io**,
with a small **REST** surface for initial loads and an optional **Discord bot**
(LLM-assisted) for querying status from chat.

The office has **3 rooms** (Drawing Room, Work Room 1, Work Room 2), each with
**2 fans + 3 lights** — **15 devices total**. It is read-only: the backend only
publishes state, it never accepts device commands.

## Architecture

Data flows in one direction, and modules are decoupled through observers:

```
config (env) + constants + domain types
        │
   DeviceStore ── single source of truth (onChange observer)
        ▲                       │ notifies
        │ applyUpdates          ▼
 SimulationEngine        Socket layer ──► devicesUpdated / powerUpdated / alertCreated
 (every 5s)                    │ reads
                       PowerService + AlertEngine (pure / isolated)
                                    │ onAlert observer
                                    ▼
                             Discord bot (optional) ──► posts live alerts
        │
   REST modules (device / power / alert) ── thin controllers ─► services ─► store
```

- **DeviceStore** is the only place device state lives. Reads return copies;
  mutations go through `applyUpdates` and notify subscribers via `onChange`.
- **SimulationEngine** writes to the store on an interval; it knows nothing about
  transport. Replaceable by real devices/MQTT.
- **PowerService** and **AlertEngine** are pure/isolated; controllers never
  compute — they call services.
- The **Socket layer** is the composition point that wires store → power → alert
  → broadcast. The **Discord bot** subscribes to the same alert observer.

## Project Structure

```
src/
  app.ts                       # Express app: middleware, routes, error handling
  server.ts                    # HTTP server + Socket.io + simulation + bot bootstrap
  config/
    index.ts                   # validated environment configuration
    constants.ts               # office composition, power ratings, thresholds
  core/
    errors/AppError.ts
    middleware/{globalErrorHandler,notFound,validateRequest}.ts
    utils/{catchAsync,sendResponse,logger}.ts
  store/
    deviceSeeder.ts            # builds the 15 devices
    deviceStore.ts             # in-memory SSOT + onChange observer
  simulation/
    simulation.utils.ts
    simulationEngine.ts        # start/stop, per-tick mutations
  socket/
    events.ts                  # typed event names + payloads
    socketServer.ts            # init + broadcast wiring
  modules/
    device/                    # REST read model
    power/                     # power calculations + REST
    alert/                     # alert engine + REST + onAlert observer
    usage/energyMeter.ts       # today's kWh accumulator
    bot/                       # Discord bot (commands, LLM, live alerts)
  routes/index.ts              # mounts all module routes under /api
  types/domain.ts              # Device, Alert, room/type/status unions
```

cd backend

## Environment Variables

| Variable                   | Required | Default                 | Description                                              |
| -------------------------- | -------- | ----------------------- | -------------------------------------------------------- |
| `NODE_ENV`                 | no       | `development`           | `production` hides error stack traces                    |
| `PORT`                     | no       | `6006`                  | HTTP + Socket.io port                                    |
| `CLIENT_ORIGIN`            | no       | `http://localhost:5173` | Allowed CORS / Socket.io origin (the frontend)           |
| `DISCORD_BOT_TOKEN`        | no       | _(empty)_               | Bot token. **Empty disables the bot entirely.**          |
| `DISCORD_COMMAND_PREFIX`   | no       | `!`                     | Command prefix                                           |
| `DISCORD_ALERT_CHANNEL_ID` | no       | _(empty)_               | Channel for live alerts. Empty disables live alerts.     |
| `GEMINI_API_KEY`           | no       | _(empty)_               | Gemini key. Empty → bot replies with deterministic text. |
| `LLM_MODEL`                | no       | `gemini-1.5-flash`      | Gemini model name                                        |

> **Security:** `.env` is gitignored and must never be committed. Only
> `.env.example` (placeholders) is tracked. Treat the bot token and API key as
> secrets; rotate them if they are ever exposed.

## REST API

Base path: `/api`. All responses use the envelope
`{ success, message, data }` (errors: `{ success:false, message, errors? }`).

| Method | Endpoint             | Description                                  |
| ------ | -------------------- | -------------------------------------------- |
| GET    | `/`                  | Health check                                 |
| GET    | `/api/devices`       | All 15 devices with current state            |
| GET    | `/api/devices/:id`   | A single device (404 if unknown)             |
| GET    | `/api/power/summary` | Total power, per-room power, active counts   |
| GET    | `/api/alerts`        | Current alerts (newest first)                |

Device ids follow `<room>-<type>-<n>`, e.g. `drawing-fan-1`, `wr1-light-2`,
`wr2-fan-2`.

---

## Realtime (Socket.io)

Connect to the same origin/port. The server pushes:

| Event            | Payload        | When                                   |
| ---------------- | -------------- | -------------------------------------- |
| `devicesUpdated` | `Device[]`     | On connect (snapshot) and every change |
| `powerUpdated`   | `PowerSummary` | On connect and every change            |
| `alertCreated`   | `Alert`        | When a new alert is raised             |

The client sends nothing — the feed is read-only.

## Discord Bot

Optional. Disabled unless `DISCORD_BOT_TOKEN` is set.

### Commands

| Command         | Example         | Response                                                             |
| --------------- | --------------- | ------------------------------------------------------------------- |
| `!status`       | `!status`       | Full office status across all rooms                                 |
| `!room <name>`  | `!room work1`   | Status of one room. Aliases: `drawing`, `work1`, `work2`            |
| `!usage`        | `!usage`        | Current total power + today's estimated kWh                         |

With `GEMINI_API_KEY` set, replies are rephrased naturally by the LLM (grounded
on exact facts — it never invents numbers). Without it, replies are precise,
deterministic text.

### Live alerts

When `DISCORD_ALERT_CHANNEL_ID` is set, the bot posts alerts to that channel as
they trigger, e.g. `🔴 **CRITICAL** [Work Room 2] 3 device(s) are still ON after office hours.`

### One-time Discord setup

1. Go to the [Discord Developer Portal](https://discord.com/developers/applications) → **New Application**.
2. **Bot** tab → **Reset Token** → copy into `DISCORD_BOT_TOKEN`.
3. Under **Privileged Gateway Intents**, enable **Message Content Intent** (required to read `!` commands).
4. **OAuth2 → URL Generator**: scope `bot`, permissions **View Channel** + **Send Messages** + **Read Message History**. Open the generated URL to invite the bot to your server.
5. (Live alerts) Enable Developer Mode in Discord, right-click your alert channel → **Copy Channel ID** → set `DISCORD_ALERT_CHANNEL_ID`.

## Testing Guide

A professional, layered approach — verify each layer independently.

### 1. Static checks

```bash
npm run typecheck   # strict TS, must pass with no output
npm run build       # must exit 0
```

### 2. REST smoke test (no bot/LLM needed)

Start the server, then in another terminal:

```bash
# health
curl -s http://localhost:6006/ | jq

# all devices (expect 15)
curl -s http://localhost:6006/api/devices | jq '.data | length'

# power summary
curl -s http://localhost:6006/api/power/summary | jq '.data'

# single device
curl -s http://localhost:6006/api/devices/drawing-fan-1 | jq '.data'

# 404 path (expect success:false)
curl -s http://localhost:6006/api/devices/nope | jq

# alerts
curl -s http://localhost:6006/api/alerts | jq '.data'
```

Run `curl .../api/power/summary` a few times a few seconds apart — `totalPower`
and `activeDevices` should change, proving the simulation is mutating state.

### 3. Realtime test

Create a throwaway `socket-smoke.mjs`:

```js
import { io } from 'socket.io-client'
const socket = io('http://localhost:6006', { transports: ['websocket'] })
socket.on('connect', () => console.log('connected', socket.id))
socket.on('devicesUpdated', (d) => console.log('devices:', d.length))
socket.on('powerUpdated', (p) => console.log('power:', p.totalPower, 'W'))
socket.on('alertCreated', (a) => console.log('ALERT:', a.severity, a.message))
setTimeout(() => process.exit(0), 12000)
```

```bash
node socket-smoke.mjs
```

Expect an immediate snapshot on connect, then `devices`/`power` lines roughly
every 5 seconds. Delete the file afterward.

### 4. Discord bot test

1. Set `DISCORD_BOT_TOKEN` (+ optionally `GEMINI_API_KEY`, `DISCORD_ALERT_CHANNEL_ID`) in `.env`.
2. `npm run dev` — expect `Discord bot logged in as <name>` and either
   `Discord live alerts enabled` or `... disabled (no DISCORD_ALERT_CHANNEL_ID)`.
3. In your server, send `!status`, `!room work2`, `!usage`. Each incoming message
   logs `[Discord] message from <user>: "<text>"` — if nothing logs, the
   **Message Content Intent** is not enabled.
4. If an alert channel is configured, leave it running; alerts appear as they fire
   (edge-triggered — one message per condition, not every tick).

### 5. Verifying the alert engine on demand

Alerts are time/threshold based. To force them for a test, temporarily lower the
threshold or widen office hours in `src/config/constants.ts`
(`ALERT_THRESHOLDS.highPowerWatts`, `OFFICE_HOURS`), restart, and watch
`/api/alerts`, the socket `alertCreated` event, and the Discord channel. Revert
after testing.

## Design Notes & Future Compatibility

- **In-memory only** by design — no DB, no auth, no MQTT at this stage.
- **Swap seams:** the `SimulationEngine` shares the store's `applyUpdates` +
  `onChange` contract, so a real IoT/MQTT feed or a Prisma/Redis-backed store can
  replace a single module without touching power, alerts, sockets, or REST.
- **LLM is a phrasing layer, never the source of truth** — on any failure the bot
  falls back to exact deterministic text, so it can't emit wrong numbers.
- **Known simplifications:** alert evaluation is change-triggered (after-hours
  fires on the next device change, not the exact clock tick); "room running
  continuously" currently means "all devices on" (no duration tracking without a
  time-series store).
